## Problem Statement

Builders who use local LLMs or smaller agent harnesses lose too much time assembling a productive base setup. Useful tools exist, including gstack, Serena MCP, gbrain, graphify, and related agent infrastructure, but each has its own docs, install path, config model, and verification steps.

Botstack packages a trusted base setup into one guided installer so builders can start coding instead of spending an afternoon wiring tools together.

Botstack is not a new AI coding agent. It is the setup layer around the harness the user already chose.

## What Makes This Cool

The cool version is a terminal installer that feels careful, not magical in the bad way. It shows exactly what will be installed, where files will be written, what each tool does, how to verify success, and how to back out.

The "whoa" is: a local-agent builder runs `botstack install`, chooses the base stack, and ends with memory, MCP tools, graph/context utilities, docs, and harness-specific config working without having to learn every tool from scratch.

## Constraints

- First product shape is `botstack install`.
- Primary mode is an interactive guided installer.
- The installer must include an all-in-one option and per-tool selection.
- Documentation and install directories are first-class, not afterthoughts.
- Botstack should improve existing harnesses rather than replacing them.
- V1 should target builders who already understand agent harnesses and want a proven base setup.
- No existing repo implementation was found in this workspace.
- Normal gstack state storage under `~/.gstack` was unavailable in this environment, so this doc is saved in the workspace.

## Premises

1. Botstack is an installer and documentation toolkit, not a new AI coding agent.
2. The core promise is "trusted base setup in minutes."
3. The first audience is builders like the creator: people who already use agent harnesses and want proven leverage tools installed quickly.
4. The first product shape is `botstack install`.
5. Botstack should be tool-agnostic at the harness layer where practical.

## Landscape Awareness

Current adjacent tools mostly want to be the agent runtime or harness:

- Harnyard focuses on cross-harness configuration and symlink bridges.
- harnext packages an agent runtime, MCP, CI runner, and skills.
- Clank, Localforge, OpenCode, and altcode compete as agent experiences.
- AGNT targets production agent infrastructure.

Botstack's useful opening is different: it is the trusted starter pack around the runtime. It should install the base stack into the harness someone already uses.

EUREKA: Most projects race to ship the agent. Botstack should ship the missing setup layer around the agent.

## Approaches Considered

### Approach A: Manifest-Backed Interactive Installer

`botstack install` reads local tool manifests for gstack, Serena MCP, gbrain, graphify, and future tools. Each manifest declares the tool description, install directories, commands, config files, verification steps, and rollback notes. The installer renders this into a guided menu with "install all" and per-tool selection.

Effort: M
Risk: Med

Pros:
- Keeps every tool's install behavior explicit and reviewable.
- Lets documentation and installer share the same source of truth.
- Scales cleanly as more tools are added.

Cons:
- Requires a careful manifest format before the first tool feels polished.
- Some tools will need custom install hooks beyond declarative fields.
- Version drift across upstream tools can break installs if verification is weak.

### Approach B: Docs-First Installer Wrapper

Start with excellent docs and a thin `botstack install` wrapper that shells out to known install commands. The installer mostly acts as a guided checklist with verification prompts.

Effort: S
Risk: Low

Pros:
- Fastest path to something useful.
- Best if the immediate goal is sharing one personal base setup.
- Lower maintenance because edge cases can stay in docs initially.

Cons:
- Risks feeling like a README with a menu.
- Harder to support reliable all-in-one install over time.
- Documentation and installer behavior may drift apart.

### Approach C: Stack Profiles and Shareable Recipes

Botstack defines named profiles such as `base`, `local-llm`, `codex`, `opencode`, and `everything`. Users can install the creator's setup first, and later publish their own stack recipes.

Effort: L
Risk: Med/High

Pros:
- Most exciting long-term shape.
- Fits the goal of helping local-LLM builders keep up.
- Creates a path for community contribution.

Cons:
- Easy to overbuild before the base install is trusted.
- Recipe compatibility can get messy across OSes, harnesses, and tool versions.
- Requires stronger governance for what counts as a safe stack.

## Recommended Approach

Use Approach A: Manifest-Backed Interactive Installer.

This is the right midpoint. It gives users a careful guided setup now, but it does not trap Botstack in a pile of shell scripts. Each tool gets a manifest that describes what it is, where it installs, what it changes, how to verify it, and how to undo it. The installer becomes a renderer and executor for those manifests.

V1 should support stack profiles only lightly: `base` and `everything` can exist as built-in selections, but public recipe sharing should wait until the manifest format has survived real use.

## Product Shape

Primary command:

```sh
botstack install
```

Expected flow:

1. Detect OS, shell, package managers, and known harness directories.
2. Show a short intro: what Botstack is and what it will never do without confirmation.
3. Offer install choices:
   - Install everything
   - Install base stack
   - Choose tools individually
   - Verify existing install
   - Print plan only
4. For each selected tool, show:
   - What it does
   - Install directory
   - Config files touched
   - Commands that will run
   - Verification command
   - Rollback notes
5. Execute selected steps with logs.
6. Run verification for each tool.
7. Print a final report with successes, failures, next steps, and docs links.

## Manifest Design

Each tool manifest should include:

```yaml
id: gbrain
name: GBrain
summary: Persistent memory and search for coding agents
homepage: https://example.invalid
category: memory
install:
  default_dir: ~/.botstack/tools/gbrain
  commands:
    - description: Install CLI
      command: ...
config:
  files:
    - path: ~/.config/gbrain/config.json
      purpose: Stores local memory configuration
harnesses:
  codex:
    files:
      - path: ~/.codex/config.toml
        purpose: Registers MCP or tool integration
verify:
  commands:
    - gbrain --version
rollback:
  notes:
    - Remove ~/.botstack/tools/gbrain
    - Remove generated gbrain config block from harness config
```

The exact format can change, but the fields above are the minimum trust surface.

## Documentation Plan

Docs should be part of the product, not a separate chore.

Recommended docs:

- `README.md`: one-screen promise, quickstart, supported tools, supported harnesses.
- `docs/install-directories.md`: every directory Botstack may write to and why.
- `docs/tools/*.md`: one page per bundled tool.
- `docs/harnesses/*.md`: Codex CLI, OpenCode, Claude Code, Aider, Continue, and custom harness notes.
- `docs/verify.md`: how to prove each tool works.
- `docs/rollback.md`: how to undo each install.
- `docs/manifest-schema.md`: how to add a new tool.

The installer should link to these docs in its final report.

## Success Criteria

- A new user can run `botstack install`, choose the base stack, and finish with clear verification results.
- The installer can run in "print plan only" mode without writing files.
- Every write path is shown before execution.
- Every installed tool has a verification command.
- Every bundled tool has docs for purpose, install directory, config changes, verification, and rollback.
- Adding a new simple tool requires adding a manifest and docs, not editing installer core logic.
- The all-in-one option works as a composition of the same per-tool manifests.

## Distribution Plan

V1 distribution should be simple:

- GitHub repo with install command in the README.
- Release tarballs or a package-manager path later.
- The installer should install Botstack into a predictable directory, likely `~/.botstack`.
- Tool installs should default under `~/.botstack/tools/<tool>` unless the upstream tool has a standard location.
- Generated harness config should be marked with managed blocks so users can inspect and remove it.

Suggested initial install command:

```sh
curl -fsSL https://raw.githubusercontent.com/<owner>/botstack/main/install.sh | sh
```

This command should bootstrap only Botstack itself. It should not install the full tool stack until the interactive installer runs and the user confirms.

## Open Questions

- Which OS is v1: macOS only, Linux only, or both?
- Which harness gets first-class support first: Codex CLI, OpenCode, Claude Code, Aider, Continue, or custom MCP config?
- Which tools belong in the initial `base` stack versus the `everything` stack?
- Should Botstack install upstream tools from source, package managers, official scripts, or pinned releases?
- Should Botstack support dry-run and uninstall in v1, or only dry-run plus manual rollback notes?
- Should manifests be YAML, JSON, TOML, or executable TypeScript modules?
