## Architecture Decisions

### Platform Matrix

V1 supports macOS + Linux and multiple first-class harnesses. This increases test and config complexity, so every harness integration must have fixtures for pristine config, existing config, malformed config, and managed-block replacement.

### Config Mutation

Manifests declare typed config operations:

- `toml.merge`
- `json.merge`
- `yaml.merge` if needed by a selected harness
- `append_managed_block`
- `remove_managed_block`
- `ensure_executable`
- `copy_file`
- `mkdir`

Every write operation must support:

- dry-run rendering
- before/after plan output
- backup before mutation
- ledger entry
- verification
- rollback path

### Bootstrap

Early development uses a simple `curl | sh` bootstrap from `main`. The script may install Botstack itself and launch the interactive installer. It must not install the full tool stack without user confirmation.

This is intentionally not production-grade distribution.

### Installer State

Write local install state to:

```text
~/.botstack/state/install-ledger.jsonl
```

Step states:

```text
planned -> skipped | applied -> verified
                 \-> failed -> rollback_planned -> rolled_back | rollback_failed
```

Each ledger entry should include:

- timestamp
- run id
- plugin id
- harness id
- operation id
- state
- target path or command
- backup path if applicable
- exit code or error summary
- duration

## Failure Modes

| Codepath | Production failure | Test required | Error handling | User-visible result |
|---|---|---|---|---|
| Environment detection | unsupported OS | unit + E2E | reject before plan | clear unsupported platform message |
| Manifest loading | invalid schema version | unit | fail before plan | clear manifest error |
| Plugin loading | plugin throws during registration | unit | isolate plugin failure | plugin disabled with reason |
| Planner | dependency order wrong | unit | validate graph before execution | plan rejection |
| `toml.merge` | malformed TOML | fixture integration | no write, report parse error | clear file path and parse issue |
| `json.merge` | existing JSON uses unexpected shape | fixture integration | no write unless schema matches | clear incompatible config message |
| Managed block append | duplicate block | fixture integration | replace managed block only | clear changed-block summary |
| Backup | backup directory unwritable | fixture integration | stop before mutation | clear permission error |
| Executor | command timeout | CLI E2E | kill process, ledger failed | timeout shown in final report |
| Verifier | installed command not found | CLI E2E | mark failed verification | next steps shown |
| Ledger | ledger write fails | unit + fixture | stop execution | clear state persistence error |
| Reporter | partial failure summary wrong | unit + E2E | derive only from ledger | honest success/failure report |

## Performance Plan

V1 executes steps sequentially with per-step timeout and live status.

Rules:

- Default command timeout: 120 seconds unless manifest overrides with a capped value.
- Verification timeout: 30 seconds unless manifest overrides with a capped value.
- Stream or summarize command output while showing current step.
- Ledger records duration and timeout state.
- No parallel config writes in v1.

## Worktree Parallelization Strategy

The implementation can be split, but only after the manifest schema contract is stable.

| Step | Modules touched | Depends on |
|---|---|---|
| Runtime scaffold + CLI command | `bin/`, `cli/`, package root | — |
| Manifest schema + plugin contracts | `core/`, `plugins/` | runtime scaffold |
| Config operations | `core/config-ops/` | manifest schema |
| Planner + stack composition | `core/planner/`, `plugins/` | manifest schema |
| Executor + ledger | `core/executor/`, `core/ledger/` | planner, config ops |
| Prompt UI + reporter | `core/prompt-ui/`, `core/reporter/` | planner |
| Docs generator + docs pages | `core/docs-generator/`, `docs/` | manifest schema |
| Test fixtures + E2E harness | `tests/`, `fixtures/` | runtime scaffold |

Parallel lanes:

- Lane A: runtime scaffold -> manifest schema -> planner.
- Lane B: test fixtures after runtime scaffold.
- Lane C: docs generator after manifest schema.
- Lane D: config operations after manifest schema.
- Lane E: executor + ledger after planner and config operations.
- Lane F: prompt UI + reporter after planner.

Execution order:

1. Build runtime scaffold and manifest schema first.
2. Launch config ops, planner, docs generator, and test fixtures in parallel.
3. Build executor/ledger after planner and config ops.
4. Build prompt UI/reporter after planner.
5. Finish with E2E tests across the integrated CLI.

Conflict flags:

- `core/` will be shared by several lanes. Keep subdirectories strict and avoid editing shared barrel exports in parallel.
- `plugins/` should be partitioned by plugin directory.
- `docs/` generation and hand-written docs may conflict unless generated reference tables have a dedicated output path.
